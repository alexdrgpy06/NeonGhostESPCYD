## 2024-05-18 - [O(1) Direct Offset Calculation in packet sniffer]
**Learning:** Replacing a linear scan of every packet byte for the `0x888E` EAPOL marker with direct O(1) offset calculations—accounting for variable 802.11 header lengths (Address 4, QoS, HT Control)—yields a massive ~19x performance improvement for encrypted data frames and a ~2.7x speedup for plaintext handshakes. This is a crucial codebase-specific bottleneck because the linear scan wasted CPU cycles on every single captured data frame. Also, encrypted frames can be safely skipped by checking the Protected bit (`packet[1] & 0x40`) because they cannot contain valid plaintext EAPOL headers, saving unnecessary processing.
**Action:** When parsing complex protocols with variable-length headers in performance-critical code paths (like a packet sniffer running on a microcontroller), calculate the exact offset based on header flags rather than using a linear search or scan. Check early-exit conditions (like encryption flags) as soon as possible.
